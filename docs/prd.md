# TodoList Desktop Pin - 产品需求文档

## 1. 项目概述

### 1.1 背景与目标

**项目背景**

用户在日常工作中需要一个随时可见的待办事项提醒工具。浏览器 Tab 或独立 App 存在切换成本，而一个始终置顶的桌面小组件可以做到"零切换、低干扰、高感知"——让用户在专注主要工作的同时，也能随时看到待办清单并完成操作。

**核心目标**

1. 提供一个轻量级桌面小组件，常驻桌面并可置顶显示
2. 支持待办事项的完整生命周期管理（增删改查、标记完成）
3. 本地持久化存储，数据安全不依赖网络

**成功指标**

- 冷启动时间 < 1 秒
- 基础操作（添加/完成/删除一条待办）耗时 < 2 秒
- 窗口内存占用 < 50 MB
- 数据在应用重启后完整保留

### 1.2 目标用户

**用户画像**

- 主要：需要在桌面多任务工作的知识工作者（开发者、设计师、产品经理、写作者）
- 次要：希望替代纸质便利贴的普通桌面用户
- 平台：Windows 11 / macOS

**使用场景**

- 上班前快速记录当天任务清单，贴在桌面角落保持可见
- 工作中途随手勾选已完成项，不中断当前工作流
- 临时想到的事情快速录入，避免遗忘
- 下班前一眼扫视今日完成率

### 1.3 技术约束

**技术栈**

- 框架：Tauri 2.x（Rust 后端 + Web 前端）
- 前端：React 18 + TypeScript
- 样式：Tailwind CSS
- 数据持久化：Tauri 内置 `tauri-plugin-store`（JSON 文件存储至系统 AppData 目录）
- 构建工具：Vite

**部署环境**

- 目标平台：Windows 11（主要），macOS 13+（次要）
- 打包形式：NSIS 安装包（Windows）/ DMG（macOS）
- 不依赖任何网络服务，全量本地运行

**性能要求**

- 冷启动 < 1 秒
- 界面渲染帧率 ≥ 60 FPS
- 本地数据读写延迟 < 50 ms
- 安装包体积 < 10 MB

**兼容性要求**

- Windows 11 64-bit
- 不需要管理员权限即可安装和运行
- 支持系统 DPI 缩放（100% / 125% / 150% / 200%）

### 2.2 功能详情

#### F-001: 桌面窗口基础行为

**描述**：小组件以无边框半透明窗口的形式显示在桌面，不阻挡正常工作，同时保持可见与可操作。

**用户故事**：作为桌面用户，我希望待办小组件始终置顶在桌面右上角，以便在不切换窗口的情况下看到和操作我的任务。

**验收标准**：
- [ ] 窗口默认尺寸 280×400px，首次运行定位于屏幕右上角（距边缘 20px）
- [ ] 窗口记忆上次关闭时的位置，重启后恢复
- [ ] 可通过标题栏拖拽移动窗口到任意位置
- [ ] 默认置顶（always on top），可通过标题栏图标按钮切换置顶开关
- [ ] 支持从窗口右下角拖拽调整尺寸，最小尺寸 200×200px，最大 500×700px
- [ ] 窗口无系统默认标题栏（frameless），使用自定义标题栏区域
- [ ] 失去焦点时窗口可选进入半透明状态（透明度可配置）

**UI/交互说明**：
- 顶部自定义标题栏高度 32px，含：应用图标（左）、"TodoList" 标题文字（左）、置顶切换按钮（右）、透明度调节按钮（右）、最小化按钮（右）、关闭按钮（右）
- 窗口主体采用毛玻璃效果（`backdrop-filter: blur`）或深色/浅色主题纯色背景
- 标题栏区域为可拖拽区域（`data-tauri-drag-region`）

---

#### F-002: 添加待办

**描述**：在输入框中输入待办内容，按回车或点击添加按钮，立即将条目插入列表顶部。

**用户故事**：作为用户，我希望能在 2 秒内完成一条待办的录入，以便不打断我的工作节奏。

**验收标准**：
- [ ] 底部或顶部固定展示一个文本输入框（placeholder: "添加待办..."）
- [ ] 按 `Enter` 键触发添加，添加后输入框自动清空并保持焦点
- [ ] 新添加的条目出现在列表顶部
- [ ] 输入内容为纯空格时不允许添加（按钮置灰/Enter 无效）
- [ ] 单条待办最大字符数 200，超出时输入框显示红色边框并禁止添加
- [ ] 添加成功后触发轻量动画（条目从顶部淡入）

**UI/交互说明**：
- 输入框位于列表底部，宽度撑满，左侧含"+"图标，右侧有"添加"按钮
- 输入框聚焦时边框高亮（主题色）

---

#### F-003: 完成/取消完成

**描述**：点击条目左侧复选框，将该待办标记为已完成（文字添加删除线，样式变灰）；再次点击可撤销。

**用户故事**：作为用户，我希望一次点击就能标记一条任务完成，以便获得即时的完成感。

**验收标准**：
- [ ] 每条待办左侧显示圆形复选框
- [ ] 点击后复选框变为勾选状态，条目文字添加删除线并降低透明度
- [ ] 再次点击恢复为未完成状态
- [ ] 状态变更后自动持久化保存
- [ ] 已完成条目沉底显示（排在未完成条目下方），或通过视觉分隔线区分

**UI/交互说明**：
- 复选框使用自定义样式，未选中为空心圆，选中为主题色实心圆带勾
- 完成状态切换附带 150ms 过渡动画

---

#### F-004: 删除待办

**描述**：鼠标悬停于条目时显示删除按钮，点击后将该条目从列表中移除。

**用户故事**：作为用户，我希望能轻松删除不再需要的任务，以便保持列表整洁。

**验收标准**：
- [ ] 鼠标悬停于条目时，右侧显示删除图标（垃圾桶或 ✕）
- [ ] 点击删除图标，条目立即从列表消失（附带淡出动画）
- [ ] 删除后自动持久化保存
- [ ] 不需要二次确认弹窗（快捷删除优先）

**UI/交互说明**：
- 删除图标仅在 hover 时出现，避免界面拥挤
- 淡出动画时长 150ms

---

#### F-005: 编辑待办

**描述**：双击条目文字区域，进入行内编辑模式，修改文字后按 Enter 或失去焦点保存，按 Escape 取消。

**用户故事**：作为用户，我希望能直接修改已有任务的描述，无需删除重建。

**验收标准**：
- [ ] 双击条目文字区域，文字变为可编辑输入框，内容预填当前文字
- [ ] 按 `Enter` 或输入框失去焦点：保存修改，退出编辑模式
- [ ] 按 `Escape`：取消修改，恢复原文字，退出编辑模式
- [ ] 编辑内容为空时按 Enter：不保存，退出编辑模式（保留原文字）
- [ ] 编辑中不触发完成/删除等操作

---

#### F-006: 本地数据持久化

**描述**：所有增删改操作完成后，立即将最新数据通过 Tauri 后端写入本地 JSON 文件，应用重启后数据完整恢复。

**用户故事**：作为用户，我希望关闭和重新打开应用后，所有待办数据原封不动，以便放心使用。

**验收标准**：
- [ ] 数据存储路径：系统 AppData 目录下 `todolist-pin/todos.json`
- [ ] 每次增删改操作后 ≤ 100ms 完成写入
- [ ] 应用启动时加载本地数据，列表渲染与数据一致
- [ ] JSON 文件格式损坏时，应用不崩溃，显示空列表并记录错误日志
- [ ] 支持至少 500 条待办数据量的正常读写

---

#### F-007: 一键清除已完成

**描述**：点击"清除已完成"按钮，批量删除所有标记为完成的条目。

**用户故事**：作为用户，我希望一键清除所有已完成任务，以便快速整理列表。

**验收标准**：
- [ ] 标题栏或列表底部提供"清除已完成"按钮，仅在存在已完成条目时可点击（否则置灰）
- [ ] 点击后所有已完成条目被移除，未完成条目保留
- [ ] 操作后自动持久化保存
- [ ] 无需二次确认弹窗

---

#### F-008: 待办分类/标签（P2）

**描述**：用户可为每条待办选择一个颜色标签（如红/橙/绿/蓝/灰，共 5 种），标签以彩色圆点显示在条目左侧。

**用户故事**：作为有多个项目要处理的用户，我希望用颜色区分不同类型的任务。

**验收标准**：
- [ ] 条目添加后，可通过右键菜单或 hover 菜单选择颜色标签
- [ ] 选定标签后，条目左侧显示对应颜色圆点
- [ ] 标签信息随数据持久化保存
- [ ] 可通过点击颜色圆点循环切换标签（包括"无标签"）

#### F-009: 截止日期提醒（P2）

**描述**：用户可为待办设置截止日期，临近（当天或明天）时高亮显示，已逾期时显示红色警告。

**用户故事**：作为有时间节点意识的用户，我希望看到哪些任务快到期了，以便提前处理。

**验收标准**：
- [ ] 通过日期选择器为条目设置截止日期（可选，默认无日期）
- [ ] 今天到期：显示橙色日期标签
- [ ] 已逾期：显示红色日期标签
- [ ] 明天到期：显示黄色日期标签
- [ ] 将来到期：显示灰色日期标签
- [ ] 截止日期信息随数据持久化保存

#### F-010: 窗口透明度调节

**描述**：用户可通过标题栏按钮或滑块调整窗口整体透明度，在 50%~100% 范围内调节，设置持久化。

**用户故事**：作为需要在小组件下方看到内容的用户，我希望降低窗口透明度，以便两者同时可见。

**验收标准**：
- [ ] 标题栏提供透明度调节入口（图标点击后展开滑块）
- [ ] 滑块范围 50%~100%，步进 5%
- [ ] 实时预览透明度效果
- [ ] 透明度设置持久化，重启后恢复

#### F-011: 系统托盘集成

**描述**：点击窗口关闭按钮时，应用最小化到系统托盘而不是退出；托盘图标右键菜单提供"显示窗口"和"退出"选项。

**用户故事**：作为用户，我希望关闭窗口后应用仍在后台运行，可随时通过托盘恢复，以便不丢失数据。

**验收标准**：
- [ ] 点击窗口关闭按钮（✕）：窗口隐藏，应用继续在托盘运行
- [ ] 系统托盘显示应用图标
- [ ] 左键单击托盘图标：切换窗口显示/隐藏
- [ ] 右键托盘图标显示菜单：「显示窗口」/ 「退出」
- [ ] 点击「退出」：保存数据后彻底退出应用

---

#### F-012: 开机自启动（P2）

**描述**：用户可在应用内开关"开机自启动"，系统重启后自动拉起小组件。

**用户故事**：作为每天使用该工具的用户，我希望开机后自动看到待办列表，以便无感融入工作流。

**验收标准**：
- [ ] 设置区域提供"开机自启动"开关（Toggle）
- [ ] 开启时，向系统注册自启动项（Windows: 注册表 / macOS: LaunchAgent）
- [ ] 关闭时，从系统移除自启动项
- [ ] 开关状态持久化保存，重启后显示当前实际状态

## 3. 非功能需求

### 3.1 性能要求

- 冷启动（从点击图标到界面可交互）< 1 秒
- 添加/删除/切换完成状态的 UI 响应 < 100ms
- 数据写入磁盘延迟 < 100ms
- 500 条数据下列表滚动帧率 ≥ 60 FPS
- 内存占用 < 50 MB（稳定运行时）

### 3.2 安全要求

- 无网络请求，所有数据完全本地存储
- 不收集任何用户数据或遥测信息
- 数据文件存储于用户 AppData 目录，非程序安装目录（避免权限问题）
- 不需要管理员权限运行

### 3.3 可用性要求

- **DPI 适配**：支持 100% / 125% / 150% / 200% 系统缩放，界面元素不变形
- **键盘操作**：核心操作（添加、编辑、删除、完成切换）均可通过键盘完成
- **最小可操作尺寸**：所有可点击目标最小 32×32px（适配触摸屏 / 高 DPI）
- **视觉主题**：支持系统跟随（跟随 Windows/macOS 的深色/浅色模式自动切换）
- **中文字体**：默认使用系统 UI 字体（微软雅黑 / PingFang SC），确保中文显示清晰

## 4. 数据模型（初步）

### 4.1 核心实体

**TodoItem（待办条目）**

| 字段        | 类型             | 说明                               |
| ----------- | ---------------- | ---------------------------------- |
| id          | string (UUID v4) | 唯一标识符                         |
| content     | string           | 待办文字内容，最长 200 字符        |
| completed   | boolean          | 是否已完成，默认 false             |
| createdAt   | number (Unix ms) | 创建时间戳                         |
| updatedAt   | number (Unix ms) | 最后更新时间戳                     |
| color       | string \| null   | 颜色标签（"red"/"orange"/"green"/"blue"/null） |
| dueDate     | string \| null   | 截止日期，ISO 8601 格式（可选）    |

**AppSettings（应用设置）**

| 字段          | 类型    | 说明                              |
| ------------- | ------- | --------------------------------- |
| windowX       | number  | 窗口 X 坐标                       |
| windowY       | number  | 窗口 Y 坐标                       |
| windowWidth   | number  | 窗口宽度                          |
| windowHeight  | number  | 窗口高度                          |
| alwaysOnTop   | boolean | 是否置顶，默认 true               |
| opacity       | number  | 窗口透明度 0.5~1.0，默认 0.9      |
| autoStart     | boolean | 是否开机自启动，默认 false        |
| theme         | "auto" \| "light" \| "dark" | 主题，默认 "auto" |

### 4.2 存储结构

本应用不使用 HTTP API，前端通过 Tauri Command（IPC）与 Rust 后端通信。

**Tauri Commands（前端调用后端）**

| Command 名称      | 参数                    | 返回值           | 说明             |
| ----------------- | ----------------------- | ---------------- | ---------------- |
| `get_todos`       | 无                      | `TodoItem[]`     | 获取所有待办     |
| `add_todo`        | `content: string`       | `TodoItem`       | 添加待办         |
| `update_todo`     | `TodoItem`              | `TodoItem`       | 更新待办（编辑/完成状态） |
| `delete_todo`     | `id: string`            | `void`           | 删除待办         |
| `clear_completed` | 无                      | `void`           | 清除已完成       |
| `get_settings`    | 无                      | `AppSettings`    | 获取设置         |
| `save_settings`   | `AppSettings`           | `void`           | 保存设置         |

### 4.3 数据流概要

```
用户操作（前端 React 组件）
    ↓ invoke('tauri_command', args)
Tauri Command Handler（Rust）
    ↓ 读/写 JSON 文件
AppData/todolist-pin/todos.json
AppData/todolist-pin/settings.json
    ↑ 返回结果
前端状态更新（React State / Zustand）
    ↑ 重新渲染列表
```

### 4.4 跨模块共享概念

| 概念        | 推荐命名    | 格式说明                        |
| ----------- | ----------- | ------------------------------- |
| 待办 ID     | `id`        | UUID v4，`xxxxxxxx-xxxx-4xxx-...` |
| 时间戳      | `createdAt` / `updatedAt` | Unix 毫秒时间戳（number）|
| 截止日期    | `dueDate`   | ISO 8601 字符串，如 `"2026-03-25"` |
| 颜色标签    | `color`     | 枚举字符串：`"red"` / `"orange"` / `"green"` / `"blue"` / `null` |

## 7. 技术方案（头脑风暴产出）

### 7.1 候选方案对比

| 维度 | 方案 A: Tauri 2.x + React + tauri-plugin-store | 方案 B: Tauri 2.x + React + 自定义 Rust IO | 方案 C: Tauri 2.x + Solid/Svelte |
|------|-----------------------------------------------|-------------------------------------------|----------------------------------|
| 概述 | 官方推荐栈，tauri-plugin-store 管理 JSON 存储 | Rust 后端自行 serde_json + fs + Mutex 锁 | 用 SolidJS/Svelte 替代 React |
| 开发复杂度 | **中** | 中高 | 中 |
| 生态成熟度 | **高**（React + Tauri 官方插件） | 高（React 生态，但存储层自建） | 中（框架生态较小） |
| 性能 | 优秀（Tauri 天然轻量） | 优秀（可精细控制 IO） | **最优**（更小 runtime） |
| 可维护性 | **高**（标准化方案，文档丰富） | 中（自建存储层需维护） | 中（社区资源较少） |
| 安装包体积 | < 5 MB | < 5 MB | **< 3 MB** |
| 团队适配 | **最佳**（React 开发者最多） | 良好 | 一般（需学习新框架） |

### 7.2 推荐方案

**选定方案 A：Tauri 2.x + React 18 + TypeScript + Tailwind CSS + tauri-plugin-store**

**选择理由**：

1. **开发效率最高**：React 生态成熟，组件库丰富，tauri-plugin-store 开箱即用
2. **风险最低**：Tauri 官方推荐的前端框架组合，文档和社区支持最好
3. **团队友好**：React 是最主流的前端框架，容易找到参考代码和解决方案
4. **性能足够**：对于 500 条待办 + 轻量 UI 的场景，React 的性能完全满足需求
5. **可扩展性**：后续可平滑升级到自定义 Rust 存储层（方案 B），无需重写前端

**放弃方案 B 的理由**：待办小组件的数据量小（<500 条），JSON 文件足够，自建 Rust IO 层属于过度工程化。若未来需要复杂查询再迁移不迟。

**放弃方案 C 的理由**：虽然 bundle 更小，但 React 的 ~40KB gzip 在桌面应用中可忽略不计，而生态劣势（UI 组件库、动画库、社区答案）会显著拖慢开发速度。

### 7.3 关键技术决策

| 决策项 | 选择 | 而非 | 理由 |
|--------|------|------|------|
| 前端框架 | React 18 | Solid/Svelte | 生态成熟度和开发效率优先 |
| 状态管理 | Zustand | Redux / Context | 轻量、零样板代码、适合小型应用 |
| 样式方案 | Tailwind CSS | CSS Modules / styled-components | 原子化 CSS 开发速度快，bundle 可 tree-shake |
| 数据存储 | tauri-plugin-store (JSON) | 自建 Rust IO / SQLite | 需求简单，官方插件开箱即用 |
| 构建工具 | Vite | Webpack | Tauri 官方推荐，HMR 快 |
| UUID 生成 | crypto.randomUUID() (前端) | uuid 库 / Rust 端生成 | 浏览器原生 API，零依赖 |
| 图标库 | Lucide React | FontAwesome / Material Icons | 轻量、tree-shakable、风格统一 |

### 7.4 风险与缓解措施

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| tauri-plugin-store 并发写入数据丢失 | 中 | 低（单窗口应用，写入频率低） | 前端防抖 300ms 合并写入；启动时校验 JSON 完整性 |
| Tauri 2.x 无边框窗口拖拽在某些 Windows 版本异常 | 高 | 中 | 使用 `data-tauri-drag-region` 官方方案；保留备用 `appWindow.startDragging()` |
| 窗口位置记忆在多显示器切换时越界 | 中 | 中 | 启动时检测屏幕边界，越界则重置到默认位置 |
| Tailwind CSS purge 误删动态类名 | 低 | 低 | 避免动态拼接类名，使用完整类名 + safelist |
| 中文路径导致 Tauri 构建失败 | 高 | 高（当前项目路径含中文） | 将项目移至英文路径，或确保 Cargo.toml 中配置 UTF-8 编码 |
